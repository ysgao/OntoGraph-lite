# Implementation Plan: Create New Ontology Entity

**Branch**: `019-create-entity` | **Date**: 2026-06-26 | **Spec**: [spec.md](spec.md)

## Summary

Implement the full "Create New Entity" workflow: a per-panel toolbar button that infers the entity type from the active panel, reads the focused entity as the parent, collects a local name from the user, constructs an IRI from the configured namespace, inserts the new entity into the in-memory model and source file, and opens the Entity Editor pre-populated with the new entity. Additionally, expose the full entity IRI as an editable field in the Entity Editor, with IRI-rename propagation across all axioms in the source file.

## Technical Context

**Language/Version**: TypeScript 5 (strict mode), Node.js 20 (extension host), Browser (webview iframe)

**Primary Dependencies**: VS Code Extension API, existing `OntologyModel`, `OntologyIndex`, `EntityEditorPanel`, `AnnotationSync`, `AxiomSync`, `FunctionalSerializer`

**Storage**: In-memory `OntologyModel` Maps; persistence via `vscode.workspace.applyEdit()` to the open document

**Testing**: Vitest (unit tests for sync/utils); VS Code Extension Test API for integration (existing pattern)

**Target Platform**: VS Code desktop (Mac/Windows/Linux)

**Project Type**: VS Code extension + webview hybrid

**Performance Goals**: Entity creation completes (model update + file write + editor open) in < 3 s; IRI rename propagates in < 5 s for ontologies up to 50,000 axioms

**Constraints**: No runtime dependency additions; no breaking changes to existing message protocol; IRI abbreviation rule must be preserved (entity IRIs use `<IRI>` bracket form in files)

**Scale/Scope**: Targets ontologies with up to 50k classes; index rebuild is acceptable at this scale

### Phase P1 — Core Entity Creation

**Goal**: Users can create any OWL entity type (class, property, individual) by clicking the "Add Entity" button in the corresponding panel toolbar. The focused entity in that panel becomes the parent via the appropriate sub-entity axiom. Entity Editor opens automatically.

#### Task P1-1: Add `ontograph.entity.defaultNamespace` setting

File: `package.json` → `contributes.configuration.properties`

Add the setting per `contracts/settings-schema.md`. Verify it appears in VS Code's Settings UI under the OntoGraph group.

#### Task P1-2: `namespaceUtils.ts` — tests first, then implement

**Test file**: `src/utils/namespaceUtils.test.ts`

Cover:
- `resolveNamespace(model, config)` → returns setting value (non-empty), else `model.metadata.iri`, else `undefined`
- `validateLocalName(name)` → `true` for valid; `{ valid: false, reason }` for empty / invalid chars
- `constructIri(namespace, localName)` → concatenates verbatim; throws if namespace does not end with `#` or `/`
- `isValidAbsoluteIri(iri)` → validates absolute IRI syntax for use in IRI rename validation

All tests must fail before implementation proceeds.

**Implementation file**: `src/utils/namespaceUtils.ts`

Local-name regex: `^[A-Za-z_][A-Za-z0-9_\-\.]*$`

#### Task P1-3: `EntityCreationSync.ts` — tests first, then implement

**Test file**: `src/sync/EntityCreationSync.test.ts`

Cover (using synthetic `.ofn` text fixtures):
- Insert a `Declaration(Class(<IRI>))` after the last existing Declaration — class with no parent
- Insert class cluster including `SubClassOf` when `entity.superClassIris` is populated
- Insert an object property cluster including `SubObjectPropertyOf` when `entity.superPropertyIris` is populated
- Insert a data property cluster including `SubDataPropertyOf` when `entity.superPropertyIris` is populated
- Insert an annotation property cluster including `SubAnnotationPropertyOf` when `entity.superPropertyIris` is populated
- Insert a named individual cluster with no parent axiom (individuals have no sub-axiom)
- Handle empty `Ontology(...)` body (no existing entities)

All tests must fail before implementation.

**Implementation file**: `src/sync/EntityCreationSync.ts`

Export:
```typescript
export function insertNewEntity(
  documentText: string,
  entity: OWLEntityUnion,   // all five entity types supported
  model: OntologyModel,
): string;
```

Parent axiom rendered per entity type:
- `OWLClass` with `superClassIris[0]` → `SubClassOf(<newIri> <parentIri>)`
- `OWLObjectProperty` with `superPropertyIris[0]` → `SubObjectPropertyOf(<newIri> <parentIri>)`
- `OWLDataProperty` with `superPropertyIris[0]` → `SubDataPropertyOf(<newIri> <parentIri>)`
- `OWLAnnotationProperty` with `superPropertyIris[0]` → `SubAnnotationPropertyOf(<newIri> <parentIri>)`
- `OWLIndividual` → no parent axiom

Strategy for `.ofn`:
1. Find the last `Declaration(` line → insert new Declaration on the next line.
2. Generate entity cluster text using `generateEntityCluster()` from `FunctionalSerializer`.
3. Find insertion point for entity cluster: after last cluster of same entity type, or before closing `)`.
4. Return modified text.

For non-`.ofn` formats: return `documentText` unchanged and surface a user-visible warning via `vscode.window.showWarningMessage('Entity creation is only supported for OWL Functional Syntax in this release.')`. Support for Manchester Syntax, Turtle, and OWL/XML is a TODO for a future release.

#### Task P1-4: Implement `addEntity` command

File: `src/commands/addEntity.ts` — replace stub body.

The command receives an `entityType: EntityType` argument supplied by the panel that triggers it (see P1-5). A shared `createEntity(entityType, context)` helper handles the full creation flow so each command registration is a one-liner.

Steps:
1. Guard: model loaded, else warn and return.
2. Resolve namespace: `resolveNamespace(activeModel, vscode.workspace.getConfiguration('ontograph'))`. If `undefined`, prompt via `showInputBox` (placeholder: `http://example.org/ontology#`); validate non-empty + ends with `#` or `/`.
3. Read selected entity IRI from the provider whose panel triggered the command. Map `entityType` to the correct tree provider and read `provider.view?.selection[0]?.iri`. Store as `parentIri: string | undefined`.
4. Local name input: `showInputBox({ prompt: 'Local name for the new entity', validateInput: name => validateLocalNameMessage(name, namespace, activeIndex) })`.
5. Construct IRI: `constructIri(namespace, localName)`.
6. Build minimal entity object for the given `entityType`:
   - `'class'` → `OWLClass` with `superClassIris: parentIri ? [parentIri] : []`
   - `'objectProperty'` → `OWLObjectProperty` with `superPropertyIris: parentIri ? [parentIri] : []`
   - `'dataProperty'` → `OWLDataProperty` with `superPropertyIris: parentIri ? [parentIri] : []`
   - `'annotationProperty'` → `OWLAnnotationProperty` with `superPropertyIris: parentIri ? [parentIri] : []`
   - `'individual'` → `OWLIndividual` with no parent field (no sub-individual axiom in OWL)
7. Read current document text; call `EntityCreationSync.insertNewEntity(text, entity, activeModel)`.
8. Apply edit via `vscode.workspace.applyEdit()`.
9. Add entity to the appropriate `activeModel` Map (`classes`, `objectProperties`, etc.).
10. Rebuild `activeIndex = new OntologyIndex(activeModel)`.
11. Call `refreshAllViews(activeModel, activeIndex)`.
12. Call `showEntityInfo(context, activeModel, iri)`.

#### Task P1-5: Wire `addEntity` to all entity panel toolbar buttons

Register one command per entity type in `package.json` `contributes.commands`, each passing the entity type as an argument. In `contributes.menus.view/title`, add the appropriate command with icon `$(add)` to each panel:

| View ID | Command | Entity type created |
|---------|---------|-------------------|
| `ontograph.classHierarchy` | `ontograph.addClass` | `'class'` |
| `ontograph.inferredHierarchy` | `ontograph.addClass` | `'class'` |
| `ontograph.objectProperties` | `ontograph.addObjectProperty` | `'objectProperty'` |
| `ontograph.dataProperties` | `ontograph.addDataProperty` | `'dataProperty'` |
| `ontograph.annotationProperties` | `ontograph.addAnnotationProperty` | `'annotationProperty'` |
| `ontograph.individuals` | `ontograph.addIndividual` | `'individual'` |

Each command in `extension.ts` calls the shared `createEntity(entityType, ...)` helper from `addEntity.ts`.

### Phase P3 — IRI Editing in Entity Editor

**Goal**: The Entity Editor shows the full IRI in an editable input. Saving a changed IRI renames the entity throughout the ontology source file.

#### Task P3-1: `IriRenameSync.ts` — tests first, then implement

**Test file**: `src/sync/IriRenameSync.test.ts`

Cover:
- Replaces `<oldIri>` with `<newIri>` in Declaration lines
- Replaces `<oldIri>` with `<newIri>` in SubClassOf, EquivalentClasses, AnnotationAssertion
- Leaves IRIs that merely contain `oldIri` as a substring but differ by brackets untouched
- Returns unchanged text if `oldIri` not found (no-op, no error)

All tests must fail before implementation.

**Implementation file**: `src/sync/IriRenameSync.ts`

Export:
```typescript
export function renameIri(
  documentText: string,
  oldIri: string,
  newIri: string,
): string;
```

Implementation: `documentText.replaceAll('<' + oldIri + '>', '<' + newIri + '>')`.

Note on formats: safe for `.ofn`, `.omn`, `.ttl`. For `.owl` (OWL/XML) IRIs appear as XML attribute values (`IRI="..."`); log a warning and skip rename for OWL/XML files until a format-aware implementation is added.

#### Task P3-2: Extend `EntityEditorMessages.ts`

Add to the discriminated union types per `contracts/entity-editor-messages.md`:
- `RenameIriMessage` (webview→extension): `{ command: 'renameIri'; currentIri: string; newIri: string }`
- `IriRenameResultMessage` (extension→webview): `{ command: 'iriRenameResult'; success: boolean; newIri?: string; error?: string }`

#### Task P3-3: Handle `renameIri` in `EntityEditorPanel.ts`

In `handleMessage()` add a `'renameIri'` case:
1. Validate `newIri`: non-empty, `isValidAbsoluteIri(newIri)`, not in `activeIndex`. Post error result if invalid.
2. Get current document text.
3. Call `IriRenameSync.renameIri(text, msg.currentIri, msg.newIri)`.
4. Apply edit via `vscode.workspace.applyEdit()`.
5. Update `activeModel`: for the entity's type map, delete `msg.currentIri` and re-insert with key `msg.newIri` (update `entity.iri` field).
6. Rebuild `activeIndex`.
7. Call `refreshAllViews()`.
8. Post `IriRenameResultMessage({ success: true, newIri: msg.newIri })`.

For OWL/XML format: post `IriRenameResultMessage({ success: false, error: 'IRI rename is not supported for OWL/XML format; convert to OWL Functional Syntax first.' })`.

#### Task P3-4: Make IRI field editable in `EntityEditorApp.ts`

1. Replace `<span id="entity-iri">` with `<input id="entity-iri" type="text" class="iri-input" aria-label="Entity IRI">`.
2. On `loadEntity`: set `iriInput.value = msg.iri`; store `currentIri = msg.iri`.
3. On `blur` and `keydown` (Enter): if `iriInput.value !== currentIri`, post `RenameIriMessage({ currentIri, newIri: iriInput.value })`.
4. On `iriRenameResult`:
   - Success: `currentIri = result.newIri`; `iriInput.value = result.newIri`; clear any error display.
   - Failure: `iriInput.value = currentIri`; show inline error element beneath the field.

## Risk Notes

- **IRI rename in OWL/XML format**: OWL/XML IRIs appear as XML attribute strings, not `<IRI>` bracket form. Mitigation: skip rename for OWL/XML in P3-3 and return a clear user-facing error.
- **Concurrent edits**: `applyEdit()` is version-checked. Mitigation: re-read the document text immediately before the `applyEdit()` call, inside the same async function.
- **Large ontologies and index rebuild**: Full rebuild is O(n) in entity count; measured at < 200ms for 50k entities. No mitigation needed.

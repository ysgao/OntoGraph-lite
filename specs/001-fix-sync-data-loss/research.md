# Research: Fix Spurious OWL File Changes on Sync

**Branch**: `001-fix-sync-data-loss` | **Phase**: 0 | **Date**: 2026-05-15

## Root Cause

### Decision
The spurious diff is caused by an annotation **ordering mismatch** between the in-memory model and the on-disk file. Two sync functions exhibit the flaw; a third is dead code on the same path.

### Detail

**How the model stores annotations**

`OWLEntity` stores labels in `entity.labels` (a `Record<string, string[]>` keyed by language tag) and other annotations in `entity.annotations` (a `Record<string, string[]>` keyed by property IRI). Both are plain JavaScript objects whose iteration order is key-insertion order — the order the parser populated them, which may differ from the on-disk file order.

`entityAnnotationPairs()` in `AnnotationSync.ts` always emits labels first (by language key), then other annotations (by propIri key). If the file stored them in a different order (e.g., `skos:definition` before `rdfs:label`), the generated text differs from the file even when the semantic content is identical.

---

**Bug 1 — `syncManchester` in `AnnotationSync.ts` (live path for `.omn`)**

`syncManchester` calls `generateManchesterAnnotationBlock(entity, prefixes)`, which produces the entire `Annotations:` block in model order. It then does a **full-text comparison** against the existing block. If the model order ≠ file order, the two strings differ and the function replaces the entire annotations section, reordering pre-existing annotations (violates FR-001, FR-002, FR-005).

The same block-replacement also applies when adding a new annotation: the old annotations get reordered into model order along with the new one (violates FR-003: "append without reordering pre-existing lines").

- **Live call path**: `EntityEditorPanel` → `syncAnnotationsToDocument` (format ≠ turtle) → `syncManchester`

---

**Bug 2 — `syncAxiomsTurtle` in `AxiomSync.ts` (live path for `.ttl`)**

For Turtle, `EntityEditorPanel` calls **only** `syncAxiomsToDocument`, which routes to `syncAxiomsTurtle`. That function rebuilds the entire Turtle subject block including annotation predicates. The annotation segments are generated by `entityAnnotationSegs(entity, prefixes)`, again in model order. The rebuilt block is compared byte-for-byte to the existing block. If annotation order differs, a spurious full-block rewrite is produced (violates FR-001, FR-002, FR-005).

- **Live call path**: `EntityEditorPanel` → `syncAxiomsToDocument` (turtle) → `syncAxiomsTurtle`

---

**Non-bug — `syncTurtle` in `AnnotationSync.ts`**

`AnnotationSync.syncTurtle` has the same ordering flaw as Bug 2, but it is **never called on the live path** for Turtle (EntityEditorPanel routes Turtle exclusively through AxiomSync). It is dead code for the known symptom. It should still be fixed for correctness.

---

**Functional (`.ofn`) — correct**

`syncFunctional` in `AnnotationSync.ts` uses a **key-based set diff**: it parses every `AnnotationAssertion` line for the entity, extracts a canonical `propIri|text|lang` key, then computes additions and deletions as set differences. File-order annotations are never touched unless they are actually being removed. This is the correct pattern that the Manchester and Turtle fixes should follow.

---

| Format | Sync Function | Bug Type | Live Path |

## Fix Strategy

### Decision
Apply the same key-based diff pattern that `syncFunctional` already uses, adapted for Manchester and Turtle structure.

**Manchester fix (`syncManchester`)**:
1. Parse each annotation line within the `Annotations:` section into a canonical `propIri|value|lang` key (the same key format used by functional).
2. Build model key set from `entityAnnotationPairs(entity)`.
3. Compute `toRemove` (in file, not in model) and `toAdd` (in model, not in file).
4. If both empty → return null (truly idempotent).
5. If non-empty → delete removed lines individually, insert new annotation lines individually after the last kept line. Do not touch the ordering of kept lines.

**Turtle fix (`syncAxiomsTurtle` in AxiomSync.ts)**:
1. Split the existing block into structural segments and annotation segments (already done).
2. Parse each annotation segment into a canonical key.
3. Compute `toAdd`/`toRemove` against the model key set.
4. Rebuild the block: structural segs (from model) + file-order annotation segs (kept) + new annotation segs (appended).
5. If rebuilt block equals existing block → return null (idempotent).

Note: Structural segs (rdf:type, rdfs:subClassOf, etc.) in Turtle are always fully replaced from the model because there is no ordering ambiguity constraint for them in the spec. Only annotation ordering must be preserved.

### Alternatives Considered

- **Canonicalize model annotation order to match file order at parse time**: would require storing file-insertion order in the model, which is a larger cross-cutting change (OntologyModel, all parsers, all serializers). Rejected — out of scope, violates YAGNI (Principle II).
- **Add a sort key to OWLEntity annotations**: same problem; affects model representation. Rejected.
- **Round-trip every edit through the full serializer**: rejected — defeats the purpose of incremental sync; also violates the assumption in spec/CLAUDE.md.

## Manchester Annotation Line Parsing

Manchester annotation lines inside an `Annotations:` section have the form:
```
        <propAbbrev_or_IRI> "<value>"@<lang>
```
or without language tag. The property is either `rdfs:label` or a full `<IRI>`. To extract the key:
1. Trim leading whitespace.
2. Extract the first whitespace-delimited token as the property IRI token.
3. Resolve via prefix map.
4. Extract the literal string and language tag with the same regex used by `parseFunctionalAnnotationItem`.

The regex `/"((?:[^"\\]|\\.)*)"\s*(?:@([A-Za-z][A-Za-z0-9-]*))?/` already works for both formats.

## Manchester Axiom Sync — Ordering Assessment

`syncAxiomsManchester` generates the managed sections (`SubClassOf:`, `EquivalentTo:`, `DisjointWith:`, etc.) in the fixed order produced by `generateManchesterAxiomSections`. The idempotency check is full-text. If the file has sections in a different order (e.g., `EquivalentTo:` before `SubClassOf:`), a spurious rewrite occurs.

**Decision**: This is a secondary concern. Since `syncAxiomsManchester` **replaces** all managed sections as a block (not individual lines), the section order is determined by the generator, not the file. Addressing axiom-section ordering is deferred — it requires a more complex multi-section diff and is not part of the reported user symptom (which is specifically annotation ordering). It can be added as a follow-on if axiom reordering is observed in practice.

# Implementation Plan: Load Large Ontology Files

**Branch**: `012-load-large-ontology` | **Date**: 2026-05-27 | **Spec**: `specs/012-load-large-ontology/spec.md`
**Input**: Feature specification from `/specs/012-load-large-ontology/spec.md`

## Summary

Add a "Load Ontology File…" command and toolbar button that reads any-sized ontology directly from disk (bypassing VS Code's text-editor size limit), parses via the existing Worker Thread pipeline, and populates all OntoGraph panels. Also adds a notification for users who open large files via VS Code's standard mechanism and receive a "file too large" error. Write-back and auto-reload continue to work for large files by switching file-read paths from `openTextDocument.getText()` to `vscode.workspace.fs.readFile`.

**Language/Version**: TypeScript 5 (strict mode), Node.js 20  
**Primary Dependencies**: VS Code Extension API, `vscode.workspace.fs` (raw file I/O), existing `ParserRegistry.parseAsync`, existing `AnnotationSync` / `AxiomSync`  
**Storage**: File system (read via `vscode.workspace.fs.readFile`; write via `WorkspaceEdit` + `workspace.applyEdit`)

**Target Platform**: VS Code Extension Host (Node.js process)  
**Project Type**: VS Code Extension  
**Performance Goals**: 200 MB OWL Functional Syntax file loads within 60 seconds (SC-001). Worker Thread parse already in place for >50k classes.

**Scale/Scope**: SNOMED CT snapshot ~200 MB / ~380k classes as benchmark.

## Regression Safety

Three existing mechanisms prevent save-triggering-reload and duplicate parses. All three must remain intact after this feature's changes.

### Mechanism 1 — `reloadGuard` (suppress file-watcher reload after sync write)

`AnnotationSync.syncAnnotationsToDocument` and `AxiomSync.syncAxiomsToDocument` both call `suppressReloadFor(3000)` before applying the `WorkspaceEdit`. The file-watcher callback in `extension.ts` checks `isReloadSuppressed()` and skips the reload if true.

**Impact of this feature**: No change. `suppressReloadFor` / `isReloadSuppressed` are untouched. For large files, the file watcher is set up identically in `loadOntologyFile` (same `createFileSystemWatcher` call as `handleDocument`). A sync write to a large file still suppresses the watcher reload for 3 seconds.

### Mechanism 2 — `parsedDocVersions` (skip re-parse on matching doc version)

`handleDocument` checks `parsedDocVersions.get(key) === doc.version` and skips parsing if true. `EntityEditorPanel` calls `parsedDocVersions.set(finalDoc.uri, finalDoc.version)` after each sync write so the following `onDidSaveTextDocument` event does not trigger a re-parse.

**Impact of T9** (`reloadOntology` → `workspace.fs.readFile`): The old code called `openTextDocument` inside `reloadOntology`, which triggered `onDidOpenTextDocument` → `handleDocument` → `parsedDocVersions.set`. After the switch to `workspace.fs.readFile`, no `openTextDocument` is called, so `parsedDocVersions` is **not** updated by the reload. This is safe because of mechanism 3 (below), but must be covered by a regression test: after a reload, a subsequent `onDidSaveTextDocument` must not cause a duplicate parse if content is unchanged.

**Impact on large files**: For files not open as text docs in the editor, `onDidSaveTextDocument` is never fired by a `WorkspaceEdit` application (VS Code does not fire save events for programmatic edits on files without an open editor tab). So `parsedDocVersions.set` in `EntityEditorPanel` is a no-op for large files — harmless.

### Mechanism 3 — `rawContent` check (skip re-parse if in-memory model matches file)

`handleDocument` checks `activeModel.rawContent === content` and skips parsing if true. This is the safety net when `parsedDocVersions` is stale.

**Impact of T9**: After `reloadOntology` sets `activeModel` with `rawContent` from the newly read file bytes, any subsequent `handleDocument` call for the same file compares `doc.getText()` against `rawContent`. If equal (normal case — no further external change), parsing is skipped. This prevents double-parses even without a `parsedDocVersions` update. T9 regression test must assert this path.

**Impact of `loadOntologyFile` (T1)**: After the initial direct-disk load, `activeModel.rawContent` is set from the read bytes. If the file is also open as a VS Code text document (edge case: file is 10–50 MB, above our detection threshold but below VS Code's hard limit), `handleDocument` will be triggered and skip via `rawContent` check.

### Regression test matrix for T9

| Scenario | Expected behaviour |
|---|---|
| External edit → file watcher fires → `executeReload` → `workspace.fs.readFile` | Model updated; no duplicate parse |
| After reload, user saves file in editor (normal-sized file) | `handleDocument` skips via `rawContent` check; no re-parse |
| After reload, user edits file in editor (content changed) | `handleDocument` parses new content; model updated |
| Sync write to large file → file watcher fires | `isReloadSuppressed()` = true; no reload |
| Sync write to normal file → `onDidSaveTextDocument` fires | `parsedDocVersions` check skips re-parse |

## Complexity Tracking

No constitution violations. All complexity comes from existing `ParserRegistry.parseAsync` Worker Thread architecture (Principle IV) which is already in place.

## Key Design Decisions

All decisions are recorded in `research.md`. Summary:

- **File read API**: `vscode.workspace.fs.readFile` (not `openTextDocument`) for all direct-from-disk operations.
- **Write-back**: Programmatic `openTextDocument` + `WorkspaceEdit.applyEdit` (no new sync abstraction unless T7 reveals this fails).
- **Large-file detection**: `onDidChangeActiveTextEditor` + `doc.getText() === ''` + `stat.size > 10 MB`.
- **No new runtime deps**: All APIs are in VS Code Extension API or Node.js standard library.
- **Toolbar position**: `group: "navigation@-1"` places load button before Classify at `navigation@0`.
- **`reloadGuard` / `parsedDocVersions` / `rawContent` check**: All three reload-suppression mechanisms remain intact; see Regression Safety section above.

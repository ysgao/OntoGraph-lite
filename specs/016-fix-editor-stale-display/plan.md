# Implementation Plan: Fix Entity Editor Stale Display After Save

**Branch**: `016-fix-editor-stale-display` | **Date**: 2026-06-10 | **Spec**: [spec.md](spec.md)

## Summary

After a save, the Entity Editor displays stale pre-save axioms because `sendLoadEntity` reads `currentSnapshot` from `EntityEditHistory` before `queueSyncWrite` (which calls `recordSave` to update it) completes. Fix: synchronously call `saveHistory.updateCurrentSnapshot(freshSnapshot)` before `sendLoadEntity` on the regular-save path. See [research.md](research.md) for full root-cause analysis.

## Implementation

### The Fix (exact location)

**File**: `src/views/EntityEditorPanel.ts`, before line 746 (`sendLoadEntity`)

**Before**:
```typescript
      sendLoadEntity(p, model, msg.iri);
```

**After**:
```typescript
      // Synchronously update currentSnapshot so sendLoadEntity shows post-save state.
      // recordSave() inside queueSyncWrite will later record the full undo checkpoint
      // with deletedPositions; this interim update only fixes the display.
      if (saveHistory && !isAutoSave) {
        const freshSnapshot = buildEntityPayload(model, msg.iri);
        if (freshSnapshot) { saveHistory.updateCurrentSnapshot(freshSnapshot); }
      }
      sendLoadEntity(p, model, msg.iri);
```

### Why This Is Safe

1. **Undo stack untouched**: `updateCurrentSnapshot` only sets `this.current`; undo/redo stacks are not modified.
2. **`recordSave` still runs**: Inside `queueSyncWrite`, `recordSave(newSnapshot, deletedPositions)` still executes — the second `this.current = newSnapshot` assignment is idempotent (same fresh snapshot).
3. **No latency added**: Both `buildEntityPayload` and `updateCurrentSnapshot` are synchronous and O(entity fields).
4. **autoSave unaffected**: The `!isAutoSave` guard preserves existing autoSave behaviour.
